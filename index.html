Schema v1 · PowerShell Reference Implementation

When an agent asks a machine about itself, what shape should the answer be?

UMI is an open schema that defines how machines describe themselves to AI agents. Structured objects, predictable property names, zero parsing. Implement it in any language. The PowerShell module is the reference.

01The problem

Every AI agent that touches a machine reinvents the same parsing logic from scratch. Two agents can't hand each other system data without playing telephone.

Guess column positions Handle spaces in mount paths Convert "458G" to bytes Hope the format hasn't changed Write a different parser for Windows

Read a property. That's it. Same shape on every OS. Same property names. JSON-serializable by default.

02The schemas

Five schemas covering the fundamentals. JSON Schema 2020-12. Implementation-agnostic — build in PowerShell, Python, Rust, whatever.

Disk
Mounted volumes — capacity, usage, filesystem type
DeviceName MountPoint TotalBytes UsedBytes FreeBytes UsedPercent FileSystem
Network
Interfaces — IPs, MACs, DNS, gateway, link state
InterfaceName InterfaceType IPv4Address MacAddress DefaultGateway DnsServers
Process
Running processes — CPU, memory, owner, status
ProcessName ProcessId CpuPercent MemoryBytes Status User
Uptime
System identity — hostname, OS, arch, boot time
Hostname OS Architecture UptimeSeconds CpuCount TotalMemoryBytes
User
Local accounts — groups, admin status, home dir
Username UserId IsAdmin IsCurrentUser Groups HomeDirectory

03For AI agents

UMI is designed for machine consumption first. Flat objects, self-documenting names, constrained enums, raw bytes, ISO timestamps.

STEP 1
Agent 1 queries
Runs Get-UmiProcess -Top 5 -AsJson on the target machine. Gets back structured data.
→
STEP 2
Lossless handoff
Passes the JSON to Agent 2. Property names and types are guaranteed by the schema.
→
STEP 3
Agent 2 acts
Filters, alerts, remediates. Zero parsing. Zero ambiguity. Zero tokens wasted on interpretation.
# Agent-to-agent: identical data shape from any OS
# Windows machine Get-UmiDisk -AsJson # → {"DeviceName":"C:","TotalBytes":511060148224,"UsedPercent":67.2,...} # Linux machine Get-UmiDisk -AsJson # → {"DeviceName":"/dev/sda1","TotalBytes":491803041792,"UsedPercent":66.0,...} # Same property names. Same types. Any downstream agent works unchanged.

04Implement in any language

The schemas are the product. The PowerShell module is just the first implementation. If the output conforms, it's UMI-compatible.

powershell — reference implementation (included)
Import-Module ./UMI/UMI.psd1 # Cross-platform, same output shape everywhere Get-UmiDisk | Where-Object UsedPercent -gt 80 Get-UmiProcess -Top 5 -AsJson Get-UmiUptime
python — example implementation (community)
import psutil, json def get_umi_disk(): disks = [] for part in psutil.disk_partitions(): usage = psutil.disk_usage(part.mountpoint) disks.append({ "DeviceName": part.device, "MountPoint": part.mountpoint, "FileSystem": part.fstype.upper(), "TotalBytes": usage.total, "UsedBytes": usage.used, "FreeBytes": usage.free, "UsedPercent": round(usage.percent, 1), "IsRemovable": False, "Label": None }) return json.dumps(disks)
rust — example implementation (community)
use sysinfo::Disks; use serde::Serialize; // Output struct matches UMI disk.schema.json #[derive(Serialize)] struct UmiDisk { DeviceName: String, MountPoint: String, FileSystem: String, TotalBytes: u64, UsedBytes: u64, FreeBytes: u64, UsedPercent: f64, } fn get_umi_disks() -> Vec<UmiDisk> { let disks = Disks::new_with_refreshed_list(); disks.iter().map(|d| UmiDisk { DeviceName: d.name().to_string_lossy().into(), MountPoint: d.mount_point().display().to_string(), FileSystem: d.file_system().to_string_lossy() .to_uppercase(), TotalBytes: d.total_space(), FreeBytes: d.available_space(), UsedBytes: d.total_space() - d.available_space(), UsedPercent: /* calculate */, }).collect() }
